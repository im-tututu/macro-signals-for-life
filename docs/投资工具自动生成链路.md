# 投资工具自动生成链路

## 1. 目标

当前仓库已经能每天生成 `metric_snapshot`，下一步不应直接从宏观指标跳到“买某只 ETF / 卖某只 LOF”，而应先补一层：

`原始快照 -> 标准化底层资产 -> 可交易工具候选 -> 当日活跃工具池 -> 宏观分桶信号 -> 工具级动作`

这样做的原因是：

- 同一类宏观信号会对应多个可交易工具，不能把规则硬编码到某个基金代码上。
- 集思录里的基金名称、指数名称、市场标签会变，必须先做标准化。
- 真正可执行的工具池需要考虑活跃度、溢价、申赎状态、是否仍在交易。
- 宏观指标会持续扩充，底层资产层稳定后，新增指标只需要补“指标 -> 资产桶”的映射。

## 2. 当前代码里已经有的积木

现有仓库已经具备第一层 `raw` 数据基础：

- `raw_jisilu_etf`：场内 ETF 全量日快照
- `raw_jisilu_qdii` + `vw_qdii_latest`：QDII 的欧美 / 亚洲 / 商品三类视图
- `raw_sse_lively_bond`：上交所活跃国债榜单
- `raw_chinabond_bond_index` / `raw_csindex_bond_index` / `raw_cnindex_bond_index`：分 provider 的债券指数特征快照
- `map_jisilu_bond_index_candidate`：从 Jisilu ETF 名称中抽出的债券指数候选映射
- `metric_daily` / `metric_snapshot`：宏观指标日频与快照层

当前缺口主要有三块：

- 还没有“标准化底层资产 / 标准化工具主数据”层
- Python 侧 `analytics/signals.py` 仍未真正承接工具级信号
- `raw_jisilu_treasury` 当前库里还是空的，现券候选暂时只能先靠 `raw_sse_lively_bond`

## 3. 推荐的数据分层

建议保持仓库现有命名风格，继续用 `cfg_ / map_ / raw_ / signal_` 四层来建。

### 3.1 标准化底层资产层

新增一层“底层资产主表”，不要直接拿 Jisilu 的 `index_nm` 当唯一资产键。

建议新增：

- `cfg_underlying_asset`
  - `underlying_id`
  - `canonical_name`
  - `provider`
  - `provider_code`
  - `asset_class_lv1`：债券 / 股票 / 商品
  - `asset_class_lv2`：利率债 / 信用债 / 美股 / 港股 / 黄金 / 原油 / 铜等
  - `asset_class_lv3`
  - `region` / `country`
  - `duration_bucket`
  - `commodity_bucket`
  - `currency`
  - `is_active`

- `map_underlying_alias`
  - `alias_name`
  - `normalized_key`
  - `underlying_id`
  - `source`
  - `confidence`

这里的关键不是字段多，而是先把“同一个东西的不同叫法”收敛到一个 `underlying_id`。

### 3.2 可交易工具主表

再单独维护“工具”而不是混在底层资产表里。

建议新增：

- `cfg_instrument`
  - `instrument_id`
  - `instrument_type`：ETF / LOF / BOND
  - `exchange`
  - `fund_id` 或 `bond_id`
  - `display_name`
  - `issuer_nm`
  - `currency`
  - `market_scope`
  - `is_qdii`
  - `is_t0`
  - `manual_whitelist`
  - `manual_blacklist`
  - `is_active`

- `map_instrument_underlying`
  - `instrument_id`
  - `underlying_id`
  - `relation_type`：tracking / benchmark / proxy
  - `match_source`
  - `match_confidence`

这一层的意义是：

- 一个底层资产可以对应多个工具
- 同一个工具的“跟踪标的”未来可能修正
- 后续信号落地时要对工具操作，而不是对指数名称操作

### 3.3 工具日快照层

最后补一张“工具级当日状态”表，承接 Jisilu / SSE 每天更新的数据。

建议新增：

- `raw_instrument_snapshot`
  - `date`
  - `instrument_id`
  - `price`
  - `volume_wan`
  - `amount_yi`
  - `unit_total_yi`
  - `turnover_rt`
  - `premium_discount_rt`
  - `apply_status`
  - `redeem_status`
  - `ytm`
  - `duration`
  - `liquidity_score`
  - `is_tradeable`
  - `is_active_candidate`
  - `raw_source`

这样 `cfg_*` 只管静态语义，`raw_*` 只管日频状态，职责会很清楚。

## 4. 自动生成链条应该怎么跑

### 第一步：先生成底层资产

来源优先级建议如下：

- 债券 ETF：先用 `map_jisilu_bond_index_candidate` 作为种子，再回填三张 provider 债券指数特征表
- QDII：优先用 `vw_qdii_latest.market` + `index_nm` + `money_cd`
- 普通 ETF：主要用 `raw_jisilu_etf.index_nm`
- 国债现券：先用 `raw_sse_lively_bond`，后续再接 `raw_jisilu_treasury`

底层资产的自动归类建议：

- 债券
  - 先看 `map_jisilu_bond_index_candidate` 的 `type_lv1/2/3`
  - 再从 provider 债券指数特征表取 `dm / y`，分成 `0-3y / 3-5y / 5-10y / 10y+`
  - 国债现券直接按剩余期限或久期分桶
- 股票
  - 先按 `vw_qdii_latest.market` 分欧美 / 亚洲
  - 再用 `index_nm` 关键字识别美国科技、港股互联网、日股、印度等细分
- 商品
  - 先按 `market='commodity'`
  - 再用 `index_nm` 关键字识别黄金 / 原油 / 铜 / 综合商品

这里不要追求一步到位全自动。第一版完全可以：

- 自动归 80%
- 剩余 20% 落到 `unknown`
- 再用人工校正规则补齐

### 第二步：由底层资产生成工具候选

候选生成规则建议：

- ETF / LOF
  - 从 `raw_jisilu_etf` 最新日取全量
  - 若同一 `fund_id` 在 `vw_qdii_latest` 也存在，则优先使用 QDII 视图字段
  - 用 `index_nm -> underlying_id` 建立映射
- 国债现券
  - 从 `raw_sse_lively_bond` 最新日取前若干名
  - 每只券直接视作一个工具，同时映射到“国债现券-短 / 中 / 长久期”底层资产桶

这一层输出的不是最终可投池，而是“候选池”。

### 第三步：从候选池筛出当日活跃工具池

建议每天按底层资产桶打分，保留每个桶 1 到 3 个工具。

可用规则：

- 基础门槛
  - `amount_yi`、`volume_wan`、`unit_total_yi` 不能太低
  - QDII 若 `apply_status / redeem_status` 异常，要降权
  - 溢价率绝对值过高要降权
- 活跃度评分
  - `liquidity_score = 0.5 * log(amount_yi+1) + 0.3 * log(volume_wan+1) + 0.2 * log(unit_total_yi+1)`
- 稳定性过滤
  - 最近 N 天持续有成交
  - 名称、跟踪标的不频繁变化
  - 优先保留大厂、老产品、主流产品

最终每个资产桶只保留少量工具，例如：

- `cn_rate_bond_5_10y` -> 1 到 2 只国债 / 政金债 ETF
- `cn_credit_bond` -> 1 到 2 只信用债 ETF
- `us_equity_broad` -> 1 到 2 只纳指 / 标普 / 美股宽基 QDII
- `hk_equity_tech` -> 1 到 2 只恒生科技 / 港股互联网
- `gold` -> 1 到 2 只黄金 ETF / QDII

## 5. 宏观快照如何连到工具

不要直接做“指标 -> 基金代码”，而是分两跳：

`metric_snapshot -> asset_bucket_signal -> instrument_signal`

### 5.1 先做资产桶信号

新增一张映射表：

- `map_metric_asset_bucket`
  - `metric_code`
  - `bucket_id`
  - `effect_direction`
  - `weight`
  - `rule_type`
  - `note`

例如：

- `gov_10y`、`spread_cdb_gov_10y_bp` -> `cn_rate_bond_7_10y`
- `spread_aaa_credit_gov_5y_bp` -> `cn_credit_bond`
- `ust_10y`、`vix`、`nasdaq_100` -> `us_equity`
- `usd_cny`、`usd_broad` -> `qdii_fx_sensitive`
- `gold`、`wti`、`copper` -> 对应商品桶

这一步只输出：

- `bucket_id`
- `score`
- `action`：增持 / 减持 / 维持
- `reason_json`

### 5.2 再把桶信号分发到工具

工具信号可以简单做：

- 先取对应桶下当日 `is_active_candidate=1` 的工具
- 按 `bucket_score * match_confidence * liquidity_weight` 生成工具分数
- 同桶只输出前 1 到 3 个工具

这样同一个宏观结论不会被绑死到单一产品；当天最活跃的工具自然会被顶出来。

## 6. 当前仓库里最适合先落地的最小闭环

如果只做第一版，建议不要一次上完整交易建议系统，而是先做下面 4 件事：

1. 新建 `cfg_underlying_asset`
2. 新建 `cfg_instrument` + `map_instrument_underlying`
3. 新建 `raw_instrument_snapshot`
4. 新增一个日常 job：`build_investable_universe`

`build_investable_universe` 的职责只做这四步：

1. 读取最新 `raw_jisilu_etf / vw_qdii_latest / raw_sse_lively_bond / raw_chinabond_bond_index / raw_csindex_bond_index / raw_cnindex_bond_index`
2. 生成或更新底层资产主数据
3. 生成或更新工具主数据与映射
4. 写入当日工具快照并打活跃标签

到这一步，系统就已经能每天自动产出：

- 今天有哪些可投工具
- 每个工具属于哪个资产桶
- 每个桶最活跃的是哪几只

这一步完成后，再接 `metric_snapshot -> bucket_signal -> instrument_signal` 就会轻很多。

## 7. 需要特别注意的三个现实问题

### 7.1 债券映射一定要保留人工校正口

债券指数名称最容易出现：

- 简称不统一
- 中债 / 中证 / 国证混用
- ETF 名称和指数名称不完全一致

所以债券这条链必须保留：

- 自动候选
- 人工确认
- 再固化到 `map_underlying_alias`

不要试图第一版就全靠纯规则一次命中。

### 7.2 “活跃”不能只看当天成交额

有些工具会因为主题突发事件突然放量，但并不适合作为长期默认可投标的。建议把活跃定义成：

- 近 20 日平均成交额
- 当前规模
- 申赎状态
- 溢价稳定性

而不是只看单日 `amount_yi`。

### 7.3 现券和 ETF 不应共用同一套动作文案

例如：

- ETF 更适合“增持 / 减持 / 维持”
- 国债现券更适合“关注 / 配置 / 替换 / 暂缓”

底层信号可以共用，但最终输出给用户时，工具类型不同，文案层要分开。

## 8. 建议的实施顺序

### P0：先把工具池自动化

- 先不碰最终买卖建议
- 每天产出标准化可投池
- 每个资产桶只保留少量主工具

### P1：再做资产桶信号

- 先把 `metric_snapshot` 转成 `bucket_signal`
- 只做十几个核心桶，不要一开始铺太细

### P2：最后做工具级建议

- 按桶信号把动作分发到工具
- 输出给用户时再做“普通人可执行”的措辞收口

## 9. 一句话结论

目前最应该做的，不是继续堆更多宏观指标，也不是直接写“买哪只基金”的规则，而是先补一层：

`标准化底层资产 + 标准化工具主数据 + 每日活跃工具池`

这层一旦稳定，后面的宏观指标扩充和信号生成都会变成“补映射”，而不是每次重新做一次全链路拼接。
