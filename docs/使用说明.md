# Python 侧使用说明

本文档面向当前仓库的 `py/` 目录，简单说明如何在本地初始化、抓取原始数据、执行 backfill、做基础测试，以及排查 SQLite 原始表问题。

## 1. 环境准备

推荐使用项目本地虚拟环境：

```bash
python3 -m venv .venv
source .venv/bin/activate
python3 -m pip install -r py/requirements.txt
```

初始化 SQLite：

```bash
python3 py/scripts/init_db.py
```

默认数据库位置：

- [db.sqlite](/Users/zy/Documents/git/macro-signals-for-life-1/py/data/db.sqlite)

- [trading_days.csv](/Users/zy/Documents/git/macro-signals-for-life-1/py/data/reference/trading_days.csv)

## 7. 本地查看 SQLite

数据库文件：

- [db.sqlite](/Users/zy/Documents/git/macro-signals-for-life-1/py/data/db.sqlite)

最简单的图形界面方式：

- 使用 DB Browser for SQLite 打开该文件

```bash
sqlite3 py/data/db.sqlite
```

常用 SQL：

```sql
.tables
.schema raw_bond_curve
select count(*) from raw_bond_curve;
select min(date), max(date) from raw_bond_curve;
select date, curve from raw_bond_curve order by date desc, curve limit 20;
```

查重复业务主键：

```sql
select date, curve, count(*)
from raw_bond_curve
group by date, curve
having count(*) > 1;
```

## 8. 注意事项

- `--dry-run` 会真实执行抓取和 upsert 判断，但最后不提交事务。
- `dry-run` 成功路径不会发 Bark；正式写入成功时会发 Bark。
- `bond_curve` 白天抓当天可能为空，这并不一定是错误。
- `bond_curve backfill` 现在已接入交易日文件，但若交易日文件未来未及时更新，尾段仍可能 fallback 到自然日。
- `policy_rate`、`bond_curve` 这类源站有时会变更页面结构，出现异常时先看日志，再抽样对照源站。
- 建议正式 backfill 按月或按季度分段执行，不要一次跑过长区间。

## 9. 推荐操作顺序

第一次本地跑通，建议按这个顺序：

1. 建立 `.venv` 并安装依赖
2. 执行 `python3 py/scripts/init_db.py`
3. 执行 `python3 py/scripts/update_trading_calendar.py`
4. 先跑一条 `money_market --dry-run`
5. 再跑一段 `bond_curve backfill --dry-run`
6. 确认无误后去掉 `--dry-run`
