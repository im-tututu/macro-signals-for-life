# Python 侧使用说明

本文档面向当前仓库的 `py/` 目录，简单说明如何在本地初始化、抓取原始数据、执行 backfill、做基础测试，以及排查 SQLite 原始表问题。

## 1. 环境准备

推荐使用项目本地虚拟环境：

```bash
python3 -m venv .venv
source .venv/bin/activate
python3 -m pip install -r py/requirements.txt
```

```bash

```

- [db.sqlite](/Users/zy/Documents/git/macro-signals-for-life-1/py/data/db.sqlite)

## 2. 抓取单条 daily job

统一入口：

```bash
python3 py/scripts/run_daily_job.py --help
```

常用示例：

```bash
python3 py/scripts/run_daily_job.py money_market --dry-run
python3 py/scripts/run_daily_job.py bond_curve --dry-run
python3 py/scripts/run_daily_job.py bond_curve --date 2026-03-25 --dry-run
python3 py/scripts/run_daily_job.py overseas_macro --dry-run
python3 py/scripts/run_daily_job.py policy_rate --dry-run
python3 py/scripts/run_daily_job.py policy_rate_recent --limit 20 --dry-run
python3 py/scripts/run_daily_job.py futures --dry-run
python3 py/scripts/run_daily_job.py etf --dry-run
python3 py/scripts/run_daily_job.py life_asset --dry-run
python3 py/scripts/run_daily_job.py bond_index --index-id <INDEX_ID> --dry-run
```

去掉 `--dry-run` 就会真正写入 SQLite。

建议调度节奏：

- 国内来源建议统一放在北京时间 `22:10-22:40`
- 美国来源建议统一放在北京时间 `07:05-07:35`
- 更适合国内晚间窗口的 job：`money_market`、`bond_curve`、`bond_index`、`policy_rate`、`futures`、`etf`、`life_asset`
- 更适合美国清晨窗口的 job：`overseas_macro`
- 补采 / 人工型任务：`policy_rate_recent`、`trading_days_update`

Python 侧也提供了结构化 registry，可供后续 cron / 调度器直接读取：

- [registry.py](/Users/zy/Documents/git/macro-signals-for-life-1/py/src/jobs/registry.py)

也可以直接按分组运行：

```bash
python3 py/scripts/run_job_group.py cn_night --dry-run
python3 py/scripts/run_job_group.py us_morning --dry-run
python3 py/scripts/run_job_group.py manual --dry-run
```

说明：

- `cn_night`：国内晚间任务组
- `us_morning`：美国清晨任务组
- `manual`：补采 / 人工维护任务组
- `bond_index` 默认需要 `--bond-index-id`；不传时会在分组结果里标记为跳过

例如手工跑今晚任务：

```bash
python3 py/scripts/run_job_group.py cn_night
```

例如今晚任务里附带跑若干债券指数：

```bash
python3 py/scripts/run_job_group.py cn_night \
  --bond-index-id <INDEX_ID_1> \
  --bond-index-id <INDEX_ID_2>
```

补充说明：

若集思录 `cookie` 过期，接口通常只会返回前 `20` 条。项目当前支持自动续期：

1. 安装依赖并准备浏览器

```bash
npm install
npx playwright install chromium
```

2. 在 `.env` 中配置以下变量

```bash
JISILU_REFRESH_URL=http://127.0.0.1:8787/refresh/jisilu-cookie
JISILU_REFRESH_TOKEN=your-token
JISILU_WEBHOOK_TOKEN=your-token
JISILU_USERNAME=your-login
JISILU_PASSWORD=your-password
JISILU_HEADLESS=0
JISILU_ARTIFACT_DIR=py/data/tmp/jisilu_webhook
JISILU_MANUAL_LOGIN_TIMEOUT_MS=180000
```

3. 启动本地 refresh webhook

```bash
npm run jisilu:webhook
```

说明：

- Python ETF 抓取在命中“游客仅显示前 20 条”时，会自动调用 webhook 刷新 `cookie` 并重试一次
- GAS 侧也可以复用同一个 webhook
- webhook 实现文件：`tools/jisilu_cookie_webhook.mjs`
- 若登录后仍是游客态，webhook 会在 `JISILU_ARTIFACT_DIR` 下保存截图和 HTML，便于排查验证码/风控/选择器失效
- 当 `JISILU_HEADLESS=0` 时，webhook 会在自动登录失败后等待你手动完成登录，再自动提取 cookie
- 若站点对无头浏览器风控较严，建议日常直接使用 `JISILU_HEADLESS=0`

### 3.8 指标快照（metrics skeleton）

先把历史“指标说明”导入 `metric_registry`：

```bash

  --xlsx "/Users/zy/Downloads/宏观观察 (1).xlsx"
```


说明：

- `metric_registry` 导入源默认是 sheet `指标说明`
- `metric_daily` 导入源默认是 sheet `指标`
- `metric_registry` 业务键为 `code`
- `metric_daily` 业务键为 `date + code`
- 两个导入都支持重复执行（upsert 覆盖）

统一入口：

```bash

python3 py/scripts/run_metrics_snapshot.py
```

```bash
--as-of-date 2026-03-26

```

当前会用到三张表：

- `metric_registry`：唯一指标定义源（名称、分类、解读、用途、calc_type 等）
- `metric_daily`：指标日频时序值
- `metric_snapshot`：按 `as_of_date + code` 的纯统计快照（不重复存名称/解读字段）

计算口径：

- `metric_snapshot` 只按 `metric_registry` 的 active code 从 `metric_daily` 取数计算

## 8. 注意事项

- `--dry-run` 会真实执行抓取和 upsert 判断，但最后不提交事务。
- `dry-run` 成功路径不会发 Bark；正式写入成功时会发 Bark。
- `bond_curve` 白天抓当天可能为空，这并不一定是错误。
- `bond_curve backfill` 现在已接入交易日文件，但若交易日文件未来未及时更新，尾段仍可能 fallback 到自然日。
- `policy_rate`、`bond_curve` 这类源站有时会变更页面结构，出现异常时先看日志，再抽样对照源站。
- 建议正式 backfill 按月或按季度分段执行，不要一次跑过长区间。

## 9. 推荐操作顺序

第一次本地跑通，建议按这个顺序：

1. 建立 `.venv` 并安装依赖
2. 执行 `python3 py/scripts/init_db.py`
3. 执行 `python3 py/scripts/update_trading_calendar.py`
4. 先跑一条 `money_market --dry-run`
5. 再跑一段 `bond_curve backfill --dry-run`
6. 确认无误后去掉 `--dry-run`

## 10. 云端部署与定时（Ubuntu）

推荐模式：

- GitHub Actions：部署与手动补跑
- 服务器 `cron`：生产定时调度

### 10.1 GitHub Secrets

在仓库 `Settings -> Secrets and variables -> Actions` 新增：

- `SERVER_HOST`
- `SERVER_PORT`
- `SERVER_USER`
- `SERVER_APP_DIR`（例如项目在服务器上的绝对路径）
- `SERVER_SSH_KEY`（私钥全文）

工作流文件：

- [deploy-python.yml](/Users/zy/Documents/git/macro-signals-for-life-1/.github/workflows/deploy-python.yml)
- [run-remote-jobs.yml](/Users/zy/Documents/git/macro-signals-for-life-1/.github/workflows/run-remote-jobs.yml)

### 10.2 服务器初始化

```bash
ssh -p <PORT> <USER>@<HOST>
mkdir -p <APP_DIR>
cd <APP_DIR>
python3 -m venv .venv
source .venv/bin/activate
pip install -U pip
pip install -r py/requirements.txt
```

生产 `.env` 放在项目根目录（不要提交到 git）：

```bash
chmod 600 <APP_DIR>/.env
```

### 10.3 安装 cron

```bash
cd <APP_DIR>
bash tools/server/install_cron.sh
```

默认安装后会在北京时间执行：

- 工作日 `22:10` 跑 `cn_night`
- 工作日 `22:20` 跑 `metric_snapshot`
- 次日工作日 `07:10` 跑 `us_morning`
- 次日工作日 `07:20` 跑 `metric_snapshot`

### 10.4 本地同步 `.env` / `db.sqlite` 到同一远端路径

脚本：

- [sync_remote.sh](/Users/zy/Documents/git/macro-signals-for-life-1/tools/sync_remote.sh)
- [set_github_secrets_from_env.sh](/Users/zy/Documents/git/macro-signals-for-life-1/tools/set_github_secrets_from_env.sh)

推荐把服务器参数统一写在本地 `.env`（单一真源）：

```bash
SERVER_HOST=<HOST>
SERVER_PORT=<PORT>
SERVER_USER=<USER>
SERVER_APP_DIR=<APP_DIR>
```

同步到远端（给 `sync_remote.sh` 用）：

```bash
./tools/sync_remote.sh
```

同步到 GitHub Actions Secrets（给 workflow 用）：

```bash
bash tools/set_github_secrets_from_env.sh
```

说明：

- 上述脚本只更新 `SERVER_HOST/SERVER_PORT/SERVER_USER/SERVER_APP_DIR`
- `SERVER_SSH_KEY` 建议在 GitHub Secrets 中手工维护（通常一次即可）
- `.env` 默认不提交到仓库，因此不建议也无法通过 workflow 自动读取 `.env` 来改 Secrets
